# 风控压测说明

## 推荐接口

### 1. `POST /chat/send`
- 最推荐作为主压测接口。
- 会命中：网关限流、认证、CSRF、风控评估、Redis 高频计数、MQ 异步审计、消息落库。
- 优点：请求体简单、并发容易拉高、对风控链路覆盖最完整。

### 2. `POST /user/follow/{userId}`
- 适合做轻量级高并发压测。
- 会命中：网关限流、认证、CSRF、风控评估、Redis 频控、关注关系写库。
- 优点：业务噪音小，适合观察风控本身的吞吐与延迟。

### 3. `POST /goods`
- 适合做“重业务写路径”压测。
- 会命中：网关限流、认证、CSRF、风控评估、商品写库、审核 MQ。
- 优点：能体现升级后风控不会把重写接口进一步拖慢。

### 4. `POST /errand`
- 适合做另一条重写链路压测。
- 会命中：网关限流、认证、CSRF、风控评估、跑腿任务写库。
- 优点：可以和商品发布场景形成对照。

## 不建议作为主压测接口

### `POST /auth/login`
- 虽然会进入风控，但前面有验证码校验。
- 压测结果很容易被验证码获取、验证码错误、用户查询等前置流程干扰。
- 更适合作为“补充场景”，不适合作为论文里的主基准接口。

## 一键脚本

脚本位置：`scripts/perf/run-risk-pressure.ps1`

特点：
- 不依赖 JMeter。
- 支持 `chat / follow / goods / errand` 四种场景。
- 自动带 `X-Requested-With`。
- 如果没有显式提供 `csrf_token`，会先发一次无害 GET 请求自动取 CSRF Cookie。
- 输出总请求数、成功数、失败数、平均耗时、P50、P95、P99、吞吐量、HTTP 状态分布、业务码分布。

## 使用示例

### 预览请求模板
```powershell
pwsh -NoProfile -File .\scripts\perf\run-risk-pressure.ps1 -Scenario chat -Preview
```

### 压测聊天发送
```powershell
pwsh -NoProfile -File .\scripts\perf\run-risk-pressure.ps1 \
  -Scenario chat \
  -BaseUrl http://127.0.0.1:8090 \
  -Concurrency 50 \
  -RequestsPerWorker 100 \
  -AccessToken <你的access_token> \
  -CsrfToken <你的csrf_token> \
  -ReceiverId 2 \
  -OutputCsvPath .\perf-chat.csv
```

### 压测关注用户
```powershell
pwsh -NoProfile -File .\scripts\perf\run-risk-pressure.ps1 \
  -Scenario follow \
  -BaseUrl http://127.0.0.1:8090 \
  -Concurrency 100 \
  -RequestsPerWorker 50 \
  -AccessToken <你的access_token> \
  -CsrfToken <你的csrf_token> \
  -TargetUserId 2
```

### 压测商品发布
```powershell
pwsh -NoProfile -File .\scripts\perf\run-risk-pressure.ps1 \
  -Scenario goods \
```

## 论文建议写法

建议至少写三组结果：

1. 轻量高频接口：`/user/follow/{id}`
- 用于体现风控对高频轻写接口的吞吐提升。

2. 中等核心接口：`/chat/send`
- 用于体现 Redis 实时风控 + MQ 异步审计的综合效果。

3. 重写接口：`/goods` 或 `/errand`
- 用于体现升级后风控不会成为重业务接口的主要瓶颈。

建议记录指标：
- 并发数
- 总请求数
- 成功率
- 平均响应时间
- P95 响应时间
- 吞吐量（req/s）
- 429 比例（限流触发情况）
- 业务失败比例（风控拦截、权限失败、参数失败）

## 建议的论文结论表达

可以写成：
- 升级前，风控主要依赖同步数据库统计与同步审计写入，高并发下响应时间波动较大。
- 升级后，在线链路改为 Redis 实时判定，审计写入改为 RocketMQ 异步批量落库，显著降低了请求主路径阻塞。
- 同时，网关限流从单机内存计数升级为 Redis 分布式限流，解决了多实例场景下限流不一致问题。
